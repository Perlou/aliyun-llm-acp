# 1.大模型应用开发

- 作为开发者开发大模型应用需要进行API调用

- 使用OpenAI Python SDK的兼容模式，也就是非ChatGPT的模型地址，而是具体服务商的
  - REST over HTTPS
  - 支持接口 Chat Completions、Responses API、Embeddings、Images、Audio、Assistants/Threads/Runs、Files、Batches、Fine-tuning、Realtime

- 创建OpenAI客户端时候要传入模型base_url和api_key
  - base_url指向提供模型的地址，通义千问百炼兼容地址是https://dashscope.aliyuncs.com/compatible-mode/v1，dashscope是模型服务灵积的代号
  - api_key要用环境变量管理，避免明文硬编码

- 使用对话接口client.chat.completions的基本内容
  - 传入参数model、messages、tools、输出控制类、工具控制类
  - model：字符串，模型名称，比如千问系列的不同规格
  - messages：JSON，携带三类信息system/user/assistant
  - 返回值解析response.choices[0].message.content

- 输出控制类参数的含义和作用
  - stream=True：开启流式输出，降低等待、改善体验，打字机效果；批量生成，适合离线
  - temperature：控制随机性/创造性（低=更稳，适合代码/事实；高=更发散，适合创意）
  - top_p：控制候选词累计概率阈值，采样范围（小=更聚焦，稳；大=更多样）
  - top_k：控制候选词的数量，从前k个中选一个，k越小越稳定聚焦，越大越多样越有创意
  - seed：固定 seed 可尽可能复现相同的采样序列，但不保证
  - max_tokens：限制生成长度
  - stop：自定义停止符
  - presence_penalty：负数鼓励用词重复与呼应，正数避免用词重复而鼓励用词多样
  - frequency_penalty：负数鼓励重复，正数压制高频重复，诗和韵文常鼓励重复，而代码法条、扩展话题不鼓励重复
  - enable_search：通义千问通过extra_body参数添加检索互联网的功能

- 多轮对话与上下文管理
  - 多轮对话需要多次调用接口
  - system用于设定AI角色、边界和风格，user是用户的输入，assistant是上下文
  - 上下文越长，处理时间越长，等待时间越长，干扰信息越多，成本越高
  - 要关注tokens的用量与计费，输入和输出的计费不同
  - 要有上下文窗口意识，过长的上下文可能被截断，做历史摘要，要保障指令
  - 上下文工程：提示词工程、RAG、工具使用、记忆机制

# 2.大模型提示词工程

- 提示词框架与模板
  - 六要素：任务目标、上下文、角色、受众、样例、输出格式，按system, user, assistant来看，角色和受众属于system消息，任务目标属于user消息，上下文、样例、输出格式属于assistant消息
  - 分隔符与结构化提示：如【】、<<< >>>、###、XML 标签等，清晰界定输入与指令
  - LlamaIndex默认RAG提示词模板，可以通过接口替换自定义模板，LlamaIndex之所以要提供提示词模板是因为用户不是直接通过调用OpenAI的对话接口来传参的，而是由LlamaIndex检索完成后自己拼接的

- 提示词技巧与最佳实践
  - 角色与受众控制：同一问题在“算法工程师”“小学老师”等角色下风格不同
  - 输出格式约束：稳定 JSON/Markdown 结构化输出，便于系统对接
  - 少样本示例（Few‑shot）：提供风格与结构样例，提升一致性与稳定性
  - 思维链（CoT）：让模型“先思考后作答”，适合推理/计算类任务
  - Meta Prompting：让模型充当“提示词教练”，对初版提示词进行评审与改写
  - 参考答案驱动的多轮迭代：生成→差距分析（Critic）→改写（Optimizer）→回测，逐步逼近理想答案
  - 量化评估提示词效果：定义维度与打分器（Grader），输出结构化评分，比较不同版本

- 模型选择与推理模型
  - 通用模型 vs 推理模型（如Qwen3推理系列）：推理模型擅长多步推导、复杂任务；通用模型响应更快、成本更低
  - 何时使用推理模型：信息模糊、海量非结构化筛选、复杂代码调试/改进等
  - 对推理模型的提示词建议：简洁清晰给出任务与背景；无需强制“逐步思考”；用示例澄清抽象要求

- 工程实践与落地
  - 提示词/流程配置化，便于领域专家参与
  - 可视化应用搭建（阿里云百炼应用构建）
  - 与RAG协同：在检索环节之外，通过提示词规范生成风格、格式与约束

# 3.大模型检索增强

- 检索增强生成的流程
  - 为大模型在生成阶段注入外部知识，提升回答的准确性与可控性
  - 分检索和生成两个阶段
  - 检索阶段包括文档解析 → 文本切片 → 向量化与建立索引 → 段落召回（Retrieval）→ 重排序（Rerank）→ 生成阶段提示词
  - 选取恰当的上下文，控制成本和质量

- 流程的各个环节的功能
  - 文件解析：将多源异构文档转为可处理文本（含表格、标题、段落结构保留）
  - 文本切片（Chunking）：切片大小与重叠的权衡，保留语境与减少噪声
  - 段落召回：基于向量检索的相似度召回，top_k 等超参设置
  - 段落重排序（Rerank）：用更强的交互式匹配模型对候选片段精排，提高相关性
  - 生成阶段提示词：合并检索片段与问题的提示词模板设计与引用规范

- 构建RAG的工具和基本用法
  - 通过LlamaIndex构建RAG，LlamaIndex就是为了将外部文档高质量接入大模型而设计
  - 数据加载与解析：SimpleDirectoryReader读取目录文档
  - 建立索引：VectorStoreIndex.from_documents，指定 embedding 模型并持久化
  - 查询引擎：index.as_query_engine，设置流式/非流式输出、选择生成模型
  - 索引持久化/加载：storage_context.persist 与 load_index_from_storage，提升启动效率与可复用性。

- 评测与指标
  - RAGAS指标体系
  - 答案相关性（Answer Relevancy）、事实一致性/忠实度（Faithfulness）
  - Answer Correctness：综合“语义相似度 + 事实准确度（观点级 F1）”评估答案正确性
  - Context Recall：ground truth 观点在检索 contexts 中被支持的占比（召回覆盖）
  - Context Precision：相关 context 的占比与排序质量（信噪比与靠前程度）
  - Ragas 其它任务指标：ToolCallAccuracy（Agent 工具调用）、SQL 等价/结果差异、BLEU（传统 NLP）

- 自动化评测
  - RAG 故障定位思路
    - 区分“检索错误”与“生成错误”
    - 直接检查本次 contexts（参考资料）是否包含所需信息
    - 简易自动化判别：test_answer（是否有效回答）、test_contexts（参考信息是否有用）；局限在于幻觉与信噪比
  - RAG 自动化评测框架（RAGAS）
    - 评测方法：构造带有标准答案与证据文档的评测集；分别评测检索子系统与生成子系统；用自动化指标与人工抽检结合
    - 实操要点：准备 question/answer/ground_truth/contexts，结合 Tongyi LLM 与 DashScope Embeddings 评测
    - 指标计算原理理解：观点抽取与匹配（由 LLM 完成）、precision 累积随位次变化
    - 自定义 RAGAS 提示词（中文化、加业务示例），以提升中文/业务场景评测稳定性
  - 其它评测与可观测框架
    - TruLens：运行过程可观测与反馈函数（Groundedness/Answer Relevance/Context Relevance）
    - DeepEval：类单元测试与 CI/CD 集成（Pass/Fail 阈值），TDD 思路
  - 指标驱动的优化闭环
    - 提升 Context Recall：补全知识库；更换/优化 embedding；Query 改写
    - 提升 Context Precision：引入 Rerank；检索器参数调优；切分策略与标题/表格增强
    - 提升 Answer Correctness：优化提示词与超参（temperature 等）；换更强模型；必要时微调
    - 关注“Lost in the Middle”与“知识浓度”：上下文过长/噪声高会稀释关键信息
    - 简易脚本能做冒烟检测，但难以覆盖事实忠实度、排序质量与可视化；用于初步定位可行，非“无法用于调优”
    - 回归流程：改一个策略—>跑 Ragas—>比对三项指标—>保留优胜配置
  - 评测运营体系（生产实践）
    - 领域专家深度参与：构建高质量评测集与 Ground Truth
    - 自动化评测 + 专家抽检协同；“发现问题→优化→回归→跟踪”的闭环
    - 维持评测集覆盖真实用户意图；沉淀规则（合规性、可操作性、情绪安抚度等业务指标）

- 持续优化检索增强能力，提高精确度和召回率
  - 文档准备与结构化（影响召回上限）
    - 结构化优先：将分散信息整理为层次清晰的 Markdown（标题/列表/表格友好检索）
    - PDF/Word 解析：DashScopeParse 将文档转 Markdown，提取图片内文字；后置“润色修正”（目录层级、表格行、上下文不一致）
    - 表格内容增强：将表格转结构化文本/键值对，便于召回与引用
    - 知识与意图闭环：建立“用户意图采集—知识更新—专家验证”机制；识别“未覆盖意图”“未利用知识”“重叠区域”并针对性扩充/清理
  - 文档切片（Chunking）策略（召回质量的基础）
    - Token 切片：可控上下文长度，适合短上下文模型
    - 句子切片（默认）：保持语义完整，通用稳妥
    - 句子窗口切片（Sentence-Window Retrieval）：在命中的句子上下扩窗，补充语境，降低切片断裂带来的信息缺失，搭配 MetadataReplacementPostProcessor
    - 语义切片（Semantic）：按语义断点自适应分段
    - 自动合并检索（Auto-Merge Retrieval）：对相邻或语义连续的片段进行合并，提升整体语义完整性
    - Markdown 切片：按标题层级切分，结构化强的文档优先
    - 选型建议：先句子切片；长上下文用窗口；专业文档试语义；Markdown 文档用专用切片；对 Token 受限用 Token 切片
    - 文本分割方法对比：固定长度切片、基于语义/标题/段落的切分与混合策略，结合数据分布选择
    - 典型问题与对策 ：过长片段引入噪声 → 缩短 chunk、优化切分策略，过短片段信息不全 → 加大 chunk 或用窗口切片，主题混淆 → 扩写标题、增加元数据标签
  - 向量化与存储（Embedding 与 Vector Store）
    - 相似度度量：余弦相似度
    - 嵌入模型对比：text-embedding-v3 效果普遍优于 v2；注意批处理与最大输入长度
    - 存储选型
      - 内存向量索引：开发/教学快速迭代
      - 本地库：Milvus、Qdrant（需自运维）
      - 云服务：DashVector、Milvus 托管版，支持“向量+标量标签”混检、弹性扩缩、可观测
  - 检索前优化（提升召回的“准度”和“全度”）
    - 提问改写与扩写：用 LLM 生成更完整的查询，显式消解歧义（如“张伟”多重实体）
    - 标题改写优化：增强标题语义可检索性（统一命名、加关键词、消歧）
    - 多步查询（规划-分解）：StepDecomposeQueryTransform + MultiStepQueryEngine，将复杂问题拆解串联检索
    - HyDE（假设文档嵌入）：先生成“假想答案文档”，再以其检索真实知识，提高匹配召回
    - 标签提取与混合检索：从文档/问题中提取结构化标签（人名/部门/职位/领域等），先过滤后做向量相似
    - 用户画像与上下文扩展：结合角色/历史偏好做查询定制
    - 反问澄清：问题不完整时引导补充关键信息
    - 其他常用策略（理解即可）：多路查询改写（multi-query）、领域词表/同义词扩展、元数据过滤（如文档类型/时间）
  - 检索后优化（降低噪声、提升相关性）
    - 重排序（ReRank）：先大 TopK 召回（如 20），再用 gte-rerank 排序取 TopN（如 3），在初筛结果上进行更细粒度语义匹配，提升前列结果的准确性
    - 相似度阈值过滤：SimilarityPostprocessor 设置 cutoff 清理弱相关片段
  - 生成阶段（答案质量与可控性）
    - 模型选择：事实性问答优先 qwen-turbo/plus；大上下文问答可选 qwen-long/turbo/plus；复杂推理可用 qwen-plus/max；行业垂直可考虑法务等专用模型
    - 提示词模板要点：明确“不足即不编造”的约束，用清晰的分隔标记组织“指令区/证据区”，按问题类型切换模板（要点式、框架先行、简洁结论等）

# 4.大模型微调

- 微调的基本概念
  - 目的：用小规模、定向标注数据让预训练模型适配具体任务/领域（如小学数学解题的推理范式）
  - 场景：当提示词工程与RAG提升有限（逻辑推理、计算步骤型任务）时，采用微调
  - 预训练：大规模通用数据，自监督/无监督，学习广泛表征
  - 微调：小规模任务数据，有监督，面向具体任务性能提升
  - 全参微调：更新全部权重，效果强但显存/成本高
  - 高效微调PEFT：仅训练少量附加参数，成本显著降低，常见方法：
    - LoRA（核心考点）：以低秩分解形式训练适配矩阵A、B，不改动原权重；推理时将A·B与原权重叠加；易回退与发布（只分发小体积增量）
    - Adapter Tuning：插入适配层，仅训适配层
    - Prompt Tuning：学习可训练提示向量，冻结模型参数
  - LoRA思想：ΔW为低秩近似，仅训练A(d×r)、B(r×d)；参数量远小于W(d×d)
  - LoRA优势：参数/显存开销小，性能-成本比高，不改变基础权重，易合并/回退
  - LoRA超参r（秩）的经验：
    - 小数据集（1k–10k）：r ≤ 16，避免过度表征与过拟合
    - 大数据集（100k+）：可尝试 r ≥ 32，捕捉更复杂模式
  - 模型参数显存估算：1.5B FP32≈5.6GB；训练时常需多倍显存；LoRA可显著降低需求
  - 训练时间与费用与总Tokens、GPU吞吐、卡数和单价相关（文档给出估算示例）
  - PAI DSW GPU实例（A10 30GB示例）、ModelScope镜像、ms-swift训练框架（支持350+ LLM/90+ MLLM）
  - 基础模型示例：Qwen2.5-1.5B-Instruct

- 微调的实验与评测
  - 微调数据集构建（必考）
    - 至少1000+条优质样本对复杂任务更稳
    - 质量优先：准确、覆盖多样场景、类别平衡、可迭代完善
    - 若数据不足：可结合知识库检索（RAG）或用大模型生成样本、人工标注、外部采集
  - 训练关键超参与原理
    - learning_rate：控制步长；过大震荡/发散，过小收敛慢。可配合调度策略
    - batch_size：影响稳定性与吞吐；过大可能影响泛化
    - num_train_epochs（epoch）：数据遍历次数；过小欠拟合，过大过拟合；可早停
    - eval_steps：按步长在验证集评估，保存checkpoint与best model
    - max_length：单样本最大长度，受显存限制
    - LoRA相关：lora_rank 与数据规模匹配
  - 损失函数与优化（理解）
    - Loss/Cost：度量预测与标签差距；目标是最小化
    - 梯度下降：沿负梯度方向、配合学习率更新参数
  - 训练状态判别与调参策略（高频）
    - 训练失败：train/eval loss不降或上升，常因学习率过高→降LR
    - 欠拟合：train与eval同时下降→继续训练、增epoch/数据量、适度调参
    - 过拟合：train降但eval升→降epoch/增数据/正则化、合理r
  - 文档中的“渐进实验”要点（可直接迁移到答题）
    - 学习率0.1 → 发散（失败）→ 降到5e-5
    - 1个epoch欠拟合 → 提升到50 epoch与更大batch后出现过拟合
    - 扩容数据（100→1000+），降epoch（至3），适度增r（至8），再视情况增epoch（至15）达成收敛
    - 核心心法：小数据少学几遍，大数据可学久一点；先稳LR，再平衡epoch与数据量；r与数据规模匹配
  - 评测流程与指标（必考）
    - A/B对比：微调前后在同一测试集打分（示例为数学题正确率/格式正确性）
    - 生成任务通用指标：BLEU、ROUGE、Perplexity；RAG评测可用RAGAS（延伸）
  - 模型产物与使用（工程实务考点）
    - checkpoint管理：best_model_checkpoint、last_model_checkpoint
    - 两种部署用法：
      - 动态加载LoRA权重（体积小、增量发布便捷，需要指定基础模型）
      - 导出并融合权重（swift export --merge_lora true）得到完整模型权重，便于独立部署
  - 实操命令与框架要点（识记）
    - 训练：swift sft --learning_rate --lora_rank --num_train_epochs --dataset --batch_size --max_length --eval_step --model_type --model_id_or_path
    - 推理：加载基础模型与模板，或用Swift.from_pretrained加载LoRA权重推理
    - 导出融合：swift export --ckpt_dir ... --merge_lora true

# 5.多Agent及多模态应用

- 助理/智能体（Assistant API）基础
  - 关键对象与流程：Assistants.create（定义模型、instructions、tools）、Threads、Messages、Runs，工具调用与工具输出提交
  - Threads 是会话容器；Messages 承载消息；Runs 触发执行并跟踪状态；Assistants 定义模型、指令、工具等
  - Threads.create → Messages.create → Runs.create → Runs.wait/submit_tool_outputs
  - 工具映射：function.name <-> 本地函数（function_mapper）；工具参数 JSON Schema 约束，确保可调用性与可解析性
  - 版本与成本：按任务强度选模型（如问候用 qwen-turbo，规划/调用用 qwen-plus），控制成本与时延
- 端到端工作流
  - 单体智能体：问答 + 工具调用（NL2SQL 查询、请假 API）
    - 工具可用性提示：instructions 中描述何时调用何工具，降低幻觉调用
    - 系统提示词明确功能边界与输出格式（如“只返回列表字符串”）
    - 约束输出为结构化字符串（如 ["employee_info_agent","leave_agent"]），后用 ast.literal_eval 解析，提升可编排性与鲁棒性。
  - 多智能体编排：Planner → 执行型 Agents → Summary 汇总
    - 加意图识别的路由：在问答前判定任务类型，再选择 RAG/翻译/审查/工具
    - 复杂复合意图（查询+请假）难以一次完成；需要任务拆解与多工具编排
    - 三类意图示例与判据：公司内部文档查询、内容翻译、文档审查；仅输出类别名，便于程序分流
    - 通过意图识别控制是否注入检索上下文，避免无关召回干扰与资源浪费
    - 关键工程点：列表字符串解析、异常兜底（失败回退到 chat_agent）、在多轮间显式标注“参考信息”避免混淆
    - Planner Agent：解析用户输入，给出执行序列（列表字符串）
    - 执行 Agents：employee_info_agent（查询员工信息）、leave_agent（请假申请）、chat_agent（日常对话）
    - Summary Agent：综合上下文与各子任务结果，生成最终答复
    - 文档审查/翻译类直接 LLM 更高效；公司知识问答走 RAG
    - RAG 干扰风险：不相关段落会误导回答，需在路由前判意图
    - 百炼平台画布式多智能体编排，适合快速业务落地；与 Dify 思路相近但本地生态和对接更友好
- 调优顺序：文中提及 NL2SQL 性能可通过微调提升（如表结构、问法多样化）；优先尝试低成本手段（提示词/RAG/工具）再考虑微调（承接“逐级优化”原则）

- 多模态典型场景
  - 多模态信息提取：商品图/纸质表单/视频理解，输出结构化字段（如花纹、衣领款式）
  - AI 剧本与动画创作：多智能体工作流自动完成脚本、镜头、角色、配音/合成渲染
  - 实时音视频互动：依托低时延网络（如阿里云视频云 ARTC）构建实时语音/视频客服或助手
  - 多端集成：网页、钉钉、微信公众号、企业微信等“10分钟构建 AI 助手”系列
- 多模态系统通用链路
  - 语音助手：ASR 流式转写 → Agent/LLM 决策 → TTS 流式合成 → ARTC 实时下发
  - 图像/视频理解：视觉理解模型（必要时 OCR/版面分析）→ 结构化抽取 → 规则/Agent 后处理
  - 动画生成：需求/梗概 → 剧本 → 分镜/资产生成 → 配音/合成 → 渲染/发布（多Agent编排）

# 6.生产环境应用实践

## 6.1.性能目标与评测基线

- 明确SLO
  - TTFT（首 token 延迟）：用户首屏感受关键。
  - TPOT（每 token 生成时间）：持续生成速度关键。

- 场景侧重点
  - 对话/搜索：高 TTFT 要求；中等 TPOT。
  - 代码补全：TTFT、TPOT 都高要求。
  - 长文处理：低 TTFT、适中 TPOT。

- 基线管理与灰度
  - 设定“基线模型/历史版本”为门槛，CI/CD 中自动拦截不达标版本。
  - 定期回归测试；小流量灰度验证后全量。

## 6.2.系统性能优化（端到端降时延/提吞吐）

- 用小模型替大模型：优先选择小参或蒸馏模型，结合领域微调达到性价比最优。

- 模型压缩
  - 剪枝、量化（INT8/INT4/FP16）、知识蒸馏。

- 降低大模型计算
  - Context Cache：复用公共前缀/多轮对话上下文，显著降成本提速。
  - 批处理/去重：合并相似请求，减少往返与调度开销。

- 减少 tokens
  - 输入：只保留关键信息，长文先摘要。
  - 输出：提示词约束简短回答/结构化输出；必要时限制 max_tokens。

- 并行化
  - 数据并行、模型并行、流水线并行；大规模服务结合分片与调度。

- 不盲用 LLM
  - 硬编码标准回复、预计算、经典算法/数据结构（哈希/二分）、传统UI组件替代冗长文本。

## 6.3.用户感知优化（不改算力也提体验）

- 流式输出：显著降低感知延迟（注意负载均衡上的缓存/压缩配置）。
- 分块处理：检索与生成均可分段回传。
- 进度展示：检索/生成进度条与阶段提示。
- 错误处理与降级
  - 友好、可操作的错误文案；自动重试（限重试次数与退避间隔）。
  - 兜底降级：固定模板、规则引擎或备份模型。
- 反馈闭环：内嵌反馈入口，数据驱动迭代。

## 6.4.成本优化（与性能协同）

- 通用策略
  - 小模型替换、Context Cache、批量推理、减少 tokens、不让 LLM 处理一切。

- 计费要点（平台示例）
  - cache_token 单价约为 input_token 的 40%。
  - 批量推理（Batch）计费约为实时推理的 50%。

- 云上成本治理
  - 设预算告警，做成本分析，识别高成本模块（GPU/ECS）。
  - 高负载保障核心功能，关闭非必要高成本任务。

## 6.5.部署与验证（落地路径）

- 直接调用托管API（无需自部署）
  - 快速启动、免运维；受 QPM/TPM 限流。

- 自建推理服务（vLLM）
  - 一行命令 serve 模型（OpenAI 兼容 /v1/chat/completions,/v1/models），可 nohup 后台。
  - wrk 压测：关注 QPS、平均/尾延迟、超时错误，评估并发-时延权衡。

- 云上部署选型（按需求/能力/成本）
  - 百炼：一键/接口部署，快速上线、通义生态优先。
  - 函数计算 FC：Serverless GPU，秒级扩缩容；冷启动权衡，适合低频/异步/事件触发。
  - PAI-EAS：面向在线推理，预热、监控、蓝绿、Spot 降本。
  - ECS/ACK/ACS：完全可控/容器编排，结合 ESS/SLB 架构高可用，适合复杂依赖与成熟 DevOps。

- GPU与并发容量粗算（示例思路）
  - 显存占用 ≈ 模型权重 + KV Cache。
  - 例：DeepSeek-R1 FP8 权重约 625GB；KV Cache（MLA 优化）估算：reql(70KB)/2^30 + 625/8 < 96GB（单卡），由此推并发与上下文上限需折中。

## 6.6.稳定性与高可用

- 降资源消耗即增稳定性：小型化、缓存、异步批处理。

- 自动扩缩容与流量治理
  - ESS/FC 动态扩缩；SLB/ALB 分流与健康检查。

- 监控与告警
  - 看板：准确率、TTFT/TPOT、QPS、错误率。
  - 数据漂移检测；全链路日志与阈值告警。

- 容灾设计
  - 熔断/降级：异常即切备份模型或规则兜底。
  - 多可用区/多地域与备用环境；定期演练。

## 6.7.安全与合规（生产运行最小集合）

- 内容与行为防护
  - 输入/输出双向审查（越狱、敏感信息、幻觉）、自定义词库集中管理。
  - Agent 执行前审计、最小权限、熔断阈值。

- 基础设施安全
  - VPC/云防火墙、主机加固、KMS 静态加密、RAM 最小权限。
  - 资源耗尽型 DDoS：DDoS 高防 + WAF 访问控制 + 应用层限流与成本熔断。

- 合规要点
  - 国内算法备案；显著 AIGC 标识与用户告知；责任主体明确。

## 6.8.高频考点/易错点速记

- TTFT 影响“首屏体验”，TPOT 决定“持续输出速度”，两者优化手段不同。

- 流式输出降低“感知延迟”，不等于缩短实际计算时长。
- Context Cache 与 Batch 能同时提性能降成本；但需权衡一致性与新鲜度。
- FC 可能冷启动；EAS 有“预热”；Spot 成本低但需容错；百炼最快上生产。
- 限流与熔断是“成本与稳定”的双保险：预估计算量、限制最大 tokens/并发/重试。

D. 文本段精度更高
